            <h1><a name="p-9991635-claude-code-1" class="anchor" href="https://linux.do#p-9991635-claude-code-1"></a>如何让 Claude Code 不再进入修改地狱</h1>
<p>最近用 Claude Code 写代码，遇到了一个让人头疼的问题：前 20 分钟生成了 2 万行代码，接下来两个星期都在调试。</p>
<p>相信很多人都有类似的经历——一开始特别兴奋，让 AI 生成了完整的应用，结果在修改的时候陷入死循环。改这里，那里就坏了；越修越乱，最后只能推倒重来。</p>
<p>直到我发现了 OpenSpec，这个问题才算有了解决方案。</p>
<h2><a name="p-9991635-h-2" class="anchor" href="https://linux.do#p-9991635-h-2"></a>问题的根源</h2>
<p>回想一下，为什么会陷入修改地狱？</p>
<p>其实很简单：AI 不知道你的整体规划。每次对话都是独立的，它只能根据当前的上下文来修改代码。今天改个登录功能，明天加个权限系统，AI 根本不知道这些功能之间的关系。代码越写越多，结构越来越乱。</p>
<p>OpenSpec 的思路是：先写规范，再写代码。</p>
<h2><a name="p-9991635-claude-code-openspec-3" class="anchor" href="https://linux.do#p-9991635-claude-code-openspec-3"></a>在 Claude Code 中配置 OpenSpec</h2>
<p>OpenSpec 直接支持 Claude Code，配置很简单。</p>
<p>在项目根目录运行：</p>
<pre data-code-wrap="bash"><code class="lang-bash">openspec init
</code></pre>
<p>系统会询问你使用的 IDE，选择 Claude Code 就行。</p>
<p>初始化完成后，你会看到项目里多了个 <code>openspec/</code> 文件夹，里面有个 <code>AGENTS.md</code> 文件。这个文件告诉 AI 该怎么工作，Claude Code 会自动读取它。</p>
<h2><a name="p-9991635-proposal-apply-4" class="anchor" href="https://linux.do#p-9991635-proposal-apply-4"></a>工作流：<span class="hashtag-raw">#proposal</span> 和 <span class="hashtag-raw">#apply</span></h2>
<p>OpenSpec 的核心是两个命令的循环使用。</p>
<h3><a name="p-9991635-proposal-5" class="anchor" href="https://linux.do#p-9991635-proposal-5"></a>第一步：<span class="hashtag-raw">#proposal</span> 创建规范</h3>
<p>在 Claude Code 的聊天框里输入 <code>/openspec proposal</code>，然后描述你要做的功能。</p>
<p>AI 会在 <code>openspec/changes/功能名/</code> 下生成三个文件：</p>
<ul>
<li><code>proposal.md</code> - 功能目标和需求</li>
<li><code>tasks.md</code> - 具体任务清单</li>
<li><code>design.md</code> - 技术设计方案</li>
</ul>
<p>这时候别急着写代码，先看看这些文件。需求理解对了吗？任务拆分合理吗？设计方案有没有坑？</p>
<p>不对的地方直接改，或者重新 <code>/openspec proposal</code>。这一步多花点时间，后面能省很多事。</p>
<h3><a name="p-9991635-apply-6" class="anchor" href="https://linux.do#p-9991635-apply-6"></a>第二步：<span class="hashtag-raw">#apply</span> 开始实现</h3>
<p>规范确认没问题了，输入 <code>/openspec apply</code>。</p>
<p>AI 会严格按照 <code>tasks.md</code> 里的清单，一项一项地写代码。不再是凭感觉发挥，而是对照着任务列表干活。</p>
<p>你可以随时检查进度。发现偏离了规范，别让它继续改，重新 <code>#proposal</code> 调整任务清单就行。</p>
<h3><a name="p-9991635-archive-7" class="anchor" href="https://linux.do#p-9991635-archive-7"></a>第三步：<span class="hashtag-raw">#archive</span> 归档</h3>
<p>功能开发完，测试通过了，用 <code>/openspec archive</code> 命令把这次变更归档。</p>
<p>这个变更就会从 <code>changes/</code> 移到 <code>specs/</code>，成为项目的永久文档。下次开发新功能，AI 会参考这些历史规范，知道项目是怎么演进的。</p>
<h2><a name="p-9991635-h-8" class="anchor" href="https://linux.do#p-9991635-h-8"></a>两个文件夹的设计</h2>
<p>OpenSpec 用两个文件夹管理状态：</p>
<ul>
<li><code>specs/</code> - 已完成的规范，代表线上的真实状态</li>
<li><code>changes/</code> - 开发中的提议，代表计划中的变更</li>
</ul>
<p>这个设计很巧妙。修改已有功能时，新旧代码不会混在一起，每个变更都在独立的文件夹里，方便追踪和审查。</p>
<h2><a name="p-9991635-h-9" class="anchor" href="https://linux.do#p-9991635-h-9"></a>我的实战经验</h2>
<p>用了一段时间 OpenSpec + Claude Code，总结几个技巧：</p>
<p><strong>小步快跑</strong>：一个 <code>/proposal</code> 只做一个功能模块。完成了立即 <code>/archive</code>，再开始下一个。别贪心一次做太多，容易失控。</p>
<p><strong>及时检查</strong>：发现 Claude Code 偏离规范了，别让它继续改。停下来，重新 <code>/proposal</code> 调整任务清单。</p>
<p><strong>保留记录</strong>：<code>changes/</code> 文件夹里的文档就是你和 AI 的"合同"。出问题的时候回看规范，能快速定位原因。</p>
<h2><a name="p-9991635-h-10" class="anchor" href="https://linux.do#p-9991635-h-10"></a>从"氛围编程"到工程化</h2>
<p>用 OpenSpec 之前，我和 Claude Code 的协作更像是"氛围编程"——聊着聊着代码就出来了，但不知道为什么这么写，也不知道改了会影响什么。</p>
<p>用了 OpenSpec 之后，整个流程变得可控：</p>
<ul>
<li>通过 <code>#proposal</code> 定义清晰的规范</li>
<li>用 <code>#apply</code> 监督 AI 按规范实施</li>
<li>靠 <code>#archive</code> 积累项目知识库</li>
</ul>
<p>你不再是救火队员，而是架构师。</p>
<h2><a name="p-9991635-h-11" class="anchor" href="https://linux.do#p-9991635-h-11"></a>试试看</h2>
<p>如果你也在用 Claude Code，不妨试试 OpenSpec。</p>
<p>在项目里跑一下 <code>openspec init</code>，然后用 <code>#proposal</code> 和 <code>#apply</code> 循环工作。你会发现，AI 编程可以变得更加可控和可靠。</p>
