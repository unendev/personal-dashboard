            <p><strong>为何可以?</strong></p>
<p>OpenRouter 的 Zero Completion Insurance 只针对<strong>单个 HTTP 包</strong> 而不是全部 所以流式只有最后一个包不会计费,其他都会,所以我们应该选择非流式</p>
<p>因此,当我们收到非流式包的前半部分的时候就应该阻断,这样整个包都不会被计费</p>
<p>看起来挺简单的?</p>
<p><strong>如何阻断？</strong></p>
<p>大部分 SSL 库是不允许访问底层 SOCKS 的,而普通的 close 发出去的实际上是 FIN 而非 RST，所以我们需要自己实现一个 SSL Client</p>
<p>当然,还有另一条路 - ASM,我们完全可以 Hook SSL API 从而得到 SOCKS 对象,但是太过大便最终放弃</p>
<p>这里我使用了一个 Python 的实现,相对来说更容易自定义:</p>

<p>由于这玩意不支持证书压缩,所以还需要调整 HandshakeSetting</p>
<pre><code class="lang-auto">
tls = TLSConnection(sock)

settings = HandshakeSettings()

*# 关闭证书压缩*

settings.certificate_compression_receive = []

settings.certificate_compression_send = []

tls.handshakeClientCert(settings=settings, serverName=host, alpn=[b"http/1.1"])

</code></pre>
<p><strong>如何检测?</strong></p>
<p>首先 HTTPS 的承载并不直接是 TCP,其下面还有一层 TLS Record,在下面才是 TCP</p>
<p>在 TLS Record 处理相当简单,我们只需要对 Raw Data 查找一下前缀即可:<a href="https://github.com/tlsfuzzer/tlslite-ng/blob/02d1506badb16473faf50ebc3a413c6d789fe31f/tlslite/tlsrecordlayer.py#L1322" class="inline-onebox" rel="noopener nofollow ugc">tlslite-ng/tlslite/tlsrecordlayer.py at 02d1506badb16473faf50ebc3a413c6d789fe31f · tlsfuzzer/tlslite-ng · GitHub</a></p>
<p></p><div class="lightbox-wrapper"><a class="lightbox" href="https://linux.do/uploads/default/original/4X/2/f/2/2f2903370000393813a7c579007fb2a88c998fc3.webp" data-download-href="https://linux.do/uploads/default/2f2903370000393813a7c579007fb2a88c998fc3" title=""><img src="https://linux.do/uploads/default/original/4X/2/f/2/2f2903370000393813a7c579007fb2a88c998fc3.webp" alt="" role="presentation" data-dominant-color="F1F2F0" width="690" height="489"></a></div><p></p>
<p>只需要在这里 Hook 一下即可,在这里检测 Mark 就行了</p>
<p>然后我们就会发现,OpenRouter 像个傻逼一样,把所有数据都塞到了同一个 TLS Record,并且还不遵守 TLS Record Limit</p>
<p><strong>那么 TCP 层面呢?</strong> ，首先我们需要明白一个 TLS Record 的结构大致是：</p>
<p>±-------------±-----------------±-----------------+</p>
<p>| Record Header| Encrypted Payload| Authentication Tag|</p>
<p>±-------------±-----------------±-----------------+</p>
<p>其中末尾的 Authentication Tag 只用来验证 TLS Record 的完整性的,对于我们这种在 TCP 上工作的,<s>显然可以忽略</s> (当然实际编程中可不行忽略)</p>
<p>必须硬解了</p>
<p>TLS 1.3（AEAD：AES-GCM / ChaCha20-Poly1305)</p>
<p>TLS 1.3 记录层头部恒为 5 字节,按照以下方式定义</p>
<pre><code class="lang-auto">
struct {

ContentType type; // 1 字节

ProtocolVersion version; // 2 字节（大端）

uint16 length; // 2 字节（大端）

} TLSRecordHeader; // 总计 5 字节

</code></pre>
<p>version（2B）：<strong>legacy_version</strong></p>
<ul>
<li><strong>恒为 0x0303（看起来像 TLS 1.2）</strong> ，用于向后兼容。</li>
<li><strong>真实协商版本在握手扩展 supported_versions 里</strong> ，不是这里。</li>
</ul>
<p>length（2B）：<strong>后续片段长度（密文长度）</strong></p>
<ul>
<li>表示<strong>紧随其后的密文总字节数</strong> （<strong>含 AEAD 的 16 字节 Tag</strong> ），<strong>不包括</strong> 这 5 字节头本身。</li>
</ul>
<p>所以, <strong>加密的 TLS 1.3 记录头通常开头是 17 03 03</strong> （type=0x17, version=0x0303）,我们就可以在 TLS Record 没有完全发送过来之前,解包出部分内容所有,我们只需要魔改一下 TLSLite 的源代码,在 getNextRecord 回调我们的解码函数即可 更快！</p>
<p><strong>有趣的地方</strong></p>
<p>OpenRouter 对于 Zero Completion Insurance 似乎有一个内部 Credit，当不计费的次数超过 credit 的时候就强制计费(我的脚本理论上是 100% 不计费的)</p>
<p>和充值比例正相关,这点在文档上没有提及</p>
<p>为了防止滥用,不贴完整代码了,自行实现吧（都写到这一步了还不会）</p>
