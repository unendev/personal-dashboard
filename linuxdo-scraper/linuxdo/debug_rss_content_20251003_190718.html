          <title>【教程】用便宜的GLM4.6套餐来驱动Codex?</title>
          <dc:creator><![CDATA[zincles]]></dc:creator>
          <category>开发调优</category>
          <description><![CDATA[
            <p>穷学生，既买不起20刀起步的Codex/Claude Code，也用不起佬友们的镜像站。</p>
<p>好在GLM整了个20元一个月的套餐，新的GML4.6模型效果堪比Claude 4 sonnet，还便宜，用着不心疼。</p>
<p>但官方文档似乎没有提供接入GLM的方案，折腾了一会，确保能用了，在此编写简单的接入方案。</p>
<hr>
<p>首先你需要去智谱那边购买Coding套餐。Lite套餐首月20CNY，续费40CNY。然后申请一个API。这步就不解释了。</p>
<h1><a name="p-9149604-h-1-codexconfigtoml-1" class="anchor" href="https://linux.do#p-9149604-h-1-codexconfigtoml-1"></a>1、配置 ~/.codex/config.toml</h1>
<p>第一步是安装Codex. 不赘述。另外本文环境是在linux下。都Linux DO了，准备个linux环境不过分吧？</p>
<p>安装完毕后，打开 <code>~/.codex/config.toml</code> 文件。如果没有文件或者目录就自行新建。</p>
<p>将以下内容粘贴进去。暂时不必修改。</p>
<pre data-code-wrap="toml"><code class="lang-toml">model_provider="glm"
model="glm-4.6"  # 倘若未来出新模型了，你可以把这里的模型id改掉
model_reasoning_effort = "high"
disable_response_storage = true

[model_providers.glm]
name = "glm"
base_url = "https://open.bigmodel.cn/api/coding/paas/v4"
env_key="GLM_API_KEY" # API KEY的环境变量的名称。不需要在这里改。

</code></pre>
<hr>
<h1><a name="p-9149604-h-2-api-key-2" class="anchor" href="https://linux.do#p-9149604-h-2-api-key-2"></a>2.配置 API key 环境变量</h1>
<p>出于某些奇怪原因，似乎没法把<code>API_KEY</code>直接写进这个配置文件里。你只能在该配置文件里指定“包含了API密钥的环境变量的变量名称”。因此，我们得在其他地方把环境变量设置好。</p>
<blockquote>
<p>倘若你在其他工具里配置了含有API key的环境变量，你可以在第一步就把环境变量名称替换掉。例如，倘若你配置了Claude Code的环境变量，你可以把<code>env_key="GLM_API_KEY"</code>改成<code>env_key="ANTHROPIC_AUTH_TOKEN</code>"，这样就不需要另外设置环境变量了。这么做就可以跳过下面的步骤，直接用即可。</p>
</blockquote>
<p>打开<code>~/.bashrc</code>（或者其他初始化用的脚本），把以下文本粘贴到文件底部，从而将你的API密钥设置为环境变量。请在这里将文本替换为你的GLM API key.</p>
<pre data-code-wrap="bash"><code class="lang-bash">export GLM_API_KEY="替换为你的APIkey" # 把文本内容替换为你的APIkey
</code></pre>
<p>之后，记得 <code>source ~/.bashrc</code>，更新环境变量。</p>
<hr>
<p>完毕后，应该就能正常使用了。</p>

<h3><a href="https://nitter.net/JustinLin610/status/1973974975976808808#m" target="_blank" rel="noopener nofollow ugc">Junyang Lin (@JustinLin610)</a></h3>

  <p>People often ask about the relationships between the different Qwen models and the logic behind their naming. Currently, we have several distinct model families: LLM, Coder, VL, Omni, and a newer addition, Image. While our long-term goal is to unify...</p>


  </article>

  <div class="onebox-metadata">

  <div style="clear: both"></div>

<p>人们经常询问不同 Qwen 模型之间的关系以及它们命名背后的逻辑。目前，我们拥有几类不同的模型：LLM、Coder、VL、Omni，以及最近新增的 Image。虽然我们的长期目标是将这些模型整合成一个真正具备全面功能的统一模型，但目前我们仍将其视为独立的模型。</p>
<p>命名规则如下：</p>
<p>Qwen 目前拥有四个主要的系列：LLM（Large Language Model）、Coder、VL（Vision Language Model）和 Omni，以及新推出的 Qwen-Image。前四个系列的命名方式相当直观：我们为每个系列提供了“Instruct”和“Thinking”两种版本的模型。这些开源模型的命名遵循一定的规则，例如 Qwen3-xB-AyB-Instruct（其中“Qwen3-xB”表示模型的参数）。对于 Coder、VL 和 Omni 系列，我们只需在“Qwen3”后面加上相应的模型类型即可，例如 Qwen3-Coder-480B-A35B-Instruct。</p>
<p>我们经常进行更新。如果更新的内容并不重要，我们会在文件名称后添加年份和月份的后缀（例如：2507），从而使文件名称变为类似 Qwen3-235B-A22B-Instruct-2507 的形式。</p>
<p>Qwen3-Next的出现确实让我们感到有些意外。在花费了一年多的时间研究新的模型架构（尤其是线性注意力机制和选择性稀疏注意力机制）之后，我们终于开发出了一个原型版本，最初将其命名为“Qwen3.5”。我们迅速对这个模型进行了训练并发布了它，以帮助社区成员适应大语言模型（LLM）生态系统即将发生的变化。然而，命名问题却出现了：将其称为“Qwen3.5-Preview”似乎显得有些不妥（相关部门间甚至出现了诸如“+0.5 后又变成了 -0.4 吗？”之类的玩笑）。最终，我们决定将其命名为“Qwen3-Next”——虽然这个名称并不完美，但它更加清晰、也更能体现出我们对该模型的未来发展方向。</p>
<p>Qwen-Image 略有不同，它并不完全符合 Qwen3 系列的设计规范；因此，我们决定不为其赋予类似于 “Qwen2.5-Image” 的版本号，尽管它确实使用了 Qwen2.5-VL 中的一些组件。Qwen-Image 的扩散模型生态系统与大型语言模型（LLMs）的生态系统存在显著差异，这使得版本号的统一变得较为复杂。目前，我们对自身的架构设计及模型规模非常自信，因此我们只发布了两个版本：Qwen-Image 以及其对应的编辑工具 Qwen-Image-Edit。这样的决策可能显得有些大胆，但如果日后我们推出了规模更小或更大的变体，我们也不排斥调整模型的命名方式。</p>
<p>在 API 方面，情况相对简单。我们主要关注 large language models（大型语言模型，简称 LLMs），这些模型分为三个等级：Max、Plus 和 Flash（之前称为“Turbo”）。因此，你会看到像 Qwen3-Max 这样的名称。我们很快就会推出 Qwen3-Plus 和 Qwen3-Flash！</p>
<p>正如所说，API的命名过程确实遇到了一些问题。两年前，当我们首次推出我们的视觉语言（VL）模型时，由于过于兴奋，我们将最大的那个模型命名为“VL-Max”；但按照如今的标准来看，这个名字显然已经不再合适了。为了解决这个问题，我们刚刚发布了“Qwen3-VL-Plus”，而“Qwen3-VL-Flash”也即将推出。至于“Qwen3-VL-Max”……我们拭目以待吧。</p>
<p>在此过程中，我们还开发了专门的音频处理模型：Qwen3-ASR 和 Qwen3-TTS。我们有信心这些模型的性能处于顶尖水平——可与 GPT-4o-transcribe、GPT-4o-mini-TTS 或 Gemini-2.5-Pro 等模型相媲美——并且我们认为这类专门开发的模型对于实际应用来说至关重要。我们目前已经推出了 Qwen3-ASR-Flash 这一 API 模型，并计划在做好更多准备后，将这些模型中的部分开源出来。</p>
<p>我们非常努力地避免在命名上让用户感到困惑；但由于目前正在开发中的模型数量众多，因此一些复杂性在所难免。我们正朝着各个模型之间的“融合”方向前进，但就目前而言，这种多样性实际上有助于开发人员和用户更有效地进行实验和开发。最终，我们的目标依然很简单：提供高质量的模型，以支持您的研究并提升您的生产力。</p>
<p>对 API 模型的进一步解释：</p>
<p>有时我们会将 Plus 和 Flash 模型应用到基于公开代码（open-source）的模型上，尤其是在这些模型被开源后。然而，API 模型的发展速度通常比基于公开代码的模型快得多。因此，我们使用基于日期的“快照”（snapshots）来更新 API：每当有新的 API 版本需要发布时，我们会为这个新版本指定一个新的名称，例如 <code>qwen3-max-2025-09-23</code>。即使在新的 API 版本发布后，原始的 API（<code>qwen3-max</code>）也不会立即被替换，因为客户需要时间来适应新版本。只有在我们确信新版本没有重大问题的情况下，才会将相应的快照合并到主 API 中。</p>

          <description><![CDATA[
            <p>小玩具，一个网盘服务器，可以将对一些文件存储服务接口插件化做适配加入，多渠道存储文件，webdav协议，也提供了用户友好的简洁前端</p>
<h2><a name="p-9148132-h-1" class="anchor" href="https://linux.do#p-9148132-h-1"></a>为什么邪门？</h2>
<ul>
<li>既然是中转服务器，这个工具本身不存储文件，而是将文件提交到上游，只存储返回的链接并映射对应目录地址，下载文件时候转发回给你</li>
<li>上游接口只要满足一个要求即可适配：接收一个文件，返回文件地址（当然这个链接要能长时间有效最好）</li>
<li>默认上传存储策略：1MB以上的文件分片并发上传，分片大小策略根据接口插件配置决定，服务器存储分片链接块，下载时候多块转换回来合成一个文件链接返回客户端，可以解决上游单文件大小限制，也可以隐藏真实链接。媒体文件支持在线预览（中转服务器链接）</li>
<li>后续会有分片文件嵌入png功能并自定义分片大小来上传到上游的可选功能，并照样提供转换回来的处理</li>
</ul>
