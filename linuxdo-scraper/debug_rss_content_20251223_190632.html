            <h1><a name="p-11583241-redis-valkey-1" class="anchor" href="https://linux.do#p-11583241-redis-valkey-1" aria-label="标题链接"></a>Redis 风险评估与 Valkey 替代方案</h1>
<blockquote>
<p>这是为公司写的文档，因为这个信息差我在众多社区没有看到，担心部分佬友也不清楚，一起分享一下新信息。<br>
。<br>
国内的字节，阿里云，腾讯云等龙头企业都已经在逐步替换为ValKey<br>
而且 ValKey更像一个 “升级版真正开源的Redis” ，同时有多个龙头企业 贡献代码，性能这块甚至更强悍</p>
</blockquote>
<p><strong>前言：</strong></p>
<ul>
<li>当前的 Redis 面临两类“硬风险”：<strong>安全漏洞风险</strong>与<strong>许可证合规风险</strong>。前者可能带来“被入侵/数据泄露/服务中断”，后者可能带来“合规边界不清/授权成本不确定”。</li>
</ul>
<h2><a name="p-11583241-h-2" class="anchor" href="https://linux.do#p-11583241-h-2" aria-label="标题链接"></a>一. 背景</h2>
<ul>
<li>Redis 近期在<strong>安全漏洞暴露</strong>与<strong>许可证合规</strong>上出现持续风险；同时 Redis 在我们业务中常作为缓存、会话、分布式锁、消息等关键能力，一旦出问题会“牵一发而动全身”。</li>
<li>因此需要评估开源替代方案，降低不确定性，保障业务稳定与可持续演进。</li>
<li>业务发展因素：当前主要在本地港口使用，未来还将随海外港口陆续上线与升级。系统一旦进入海外规模化交付阶段，任何“漏洞应急/版本差异/许可证边界不清”都会被放大，形成长期经营成本与交付风险。</li>
</ul>
<h2><a name="p-11583241-redis-3" class="anchor" href="https://linux.do#p-11583241-redis-3" aria-label="标题链接"></a>二、Redis 当前核心风险分析</h2>
<h3><a name="p-11583241-h-1-4" class="anchor" href="https://linux.do#p-11583241-h-1-4" aria-label="标题链接"></a>1. 安全风险：高危漏洞与可用性风险</h3>
<ul>
<li>
<p>风险等级：<strong>高</strong>（影响面大、升级与响应成本高）。</p>
</li>
<li>
<p><strong>高危漏洞示例：CVE-2025-62507（CVSS 8.8）</strong></p>
<ul>
<li>可触发<strong>栈缓冲区溢出</strong>，在一定条件下可能导致 <strong>RCE</strong> 风险 。<br>
（存在被人“利用一个入口把门撬开”的可能，轻则异常重启，重则带来系统被入侵与数据泄露风险）</li>
<li>Redis <strong>8.2.0 及以上</strong>，<code>XACKDEL</code> 在多 ID 场景可触发问题；<strong>8.2.3 修复</strong>；可用 <code>ACL</code> 限制命令作为临时缓解。</li>
</ul>
</li>
<li>
<p><strong>DoS 漏洞示例：CVE-2025-48367（CVSS 7.5）</strong></p>
<ul>
<li><strong>未认证连接</strong>可造成重复协议错误，导致<strong>客户端饥饿</strong>并最终引发 <strong>DoS</strong>。<br>
（可能让正常业务请求排不上队，出现超时、报错甚至整体不可用）</li>
<li>修复版本（NVD）：<strong>8.0.3 / 7.4.5 / 7.2.10 / 6.2.19</strong>。</li>
</ul>
</li>
<li>
<p>业务层面的严重性 ：Redis 是“公共枢纽”，一旦被打挂/被入侵，影响往往不是单点，需要考虑系统一但失去Redis响应时，是否能在短时间内维持业务流程而不中断。</p>
</li>
<li>
<p>安全风险不是“有没有漏洞”，而是<strong>漏洞出现后我们能否在可接受时间内完成评估、升级、验证与发布</strong>；频繁高危漏洞会显著提高持续运维成本与业务暴露面。</p>
</li>
<li>
<p>短期措施：升级到修复版本；收敛网络暴露面（尽量不对公网/不对不可信网络开放）；启用并收紧 <code>ACL</code>；对高危/少用命令做最小权限控制。</p>
</li>
</ul>
<h3><a name="p-11583241-h-2-5" class="anchor" href="https://linux.do#p-11583241-h-2-5" aria-label="标题链接"></a>2. 合规风险：许可证限制收紧</h3>
<ul>
<li>
<p>风险等级：<strong>中-高</strong>（取决于公司业务形态与对外服务方式）。</p>
</li>
<li>
<p><strong>许可证变更</strong>：Redis 官方宣布自 <strong>Redis 7.4 起采用双重许可证 RSALv2 / SSPLv1</strong> 的“源码可用”模式，不再是 OSI 定义的开源许可。</p>
</li>
<li>
<p>以前的 BSD 更像“大家都能自由用”；现在更像“能看源码，但某些用法有明确限制”，一旦触碰边界可能带来合规压力与不确定成本。</p>
</li>
</ul>
<h3><a name="p-11583241-h-3-6" class="anchor" href="https://linux.do#p-11583241-h-3-6" aria-label="标题链接"></a>3. 长期发展风险：生态与路线不确定性</h3>
<ul>
<li>风险等级：<strong>中</strong>（长期影响大，但短期可通过治理与替代路径缓释）。</li>
<li><strong>社区生态分化</strong>：许可证变化导致社区分流，未来协作效率与外部贡献存在不确定性。</li>
<li><strong>功能迭代不确定性</strong>：商业导向增强可能导致开源可用能力与迭代节奏的优先级变化；企业长期可控性下降。</li>
<li>这类风险短期看不出“马上出事”，但会让我们未来在版本选择、成本与供应方依赖上更被动。</li>
</ul>
<h2><a name="p-11583241-valkey-7" class="anchor" href="https://linux.do#p-11583241-valkey-7" aria-label="标题链接"></a>三、Valkey 替代方案核心优势</h2>
<h3><a name="p-11583241-h-1-8" class="anchor" href="https://linux.do#p-11583241-h-1-8" aria-label="标题链接"></a>1. 合规性保障：开源自由</h3>
<ul>
<li><strong>许可证</strong>：Valkey 以 <strong>BSD 许可证</strong>为基础进行持续开源开发（参见 Valkey 官方历史说明）。</li>
<li><strong>治理背书</strong>：Valkey 由多方企业与社区推动，作为 <strong>Linux Foundation 项目</strong>启动（Valkey History / Linux Foundation press release）。</li>
<li>从合规角度更“省心”，不容易出现“用着用着突然要补授权/被审计卡住”的情况。</li>
</ul>
<h3><a name="p-11583241-h-2-9" class="anchor" href="https://linux.do#p-11583241-h-2-9" aria-label="标题链接"></a>2. 技术兼容性：迁移成本可控</h3>
<ul>
<li><strong>协议与客户端兼容</strong>：以 Redis 兼容协议为目标，现有常见客户端（如 Jedis、redis-py 等）通常无需改造即可连接（需在测试环境验证业务实际用到的命令与模块）。</li>
<li><strong>数据与部署兼容</strong>：可基于现有的持久化与复制策略进行迁移设计（RDB/AOF、主从、集群等），核心工作集中在“版本/参数对齐、灰度切换、回滚预案”。</li>
<li>真正的工作量在于验证、灰度和回滚，而不是大改业务代码。</li>
</ul>
<h3><a name="p-11583241-h-3-10" class="anchor" href="https://linux.do#p-11583241-h-3-10" aria-label="标题链接"></a>3. 功能与性能：在继承基础上演进</h3>
<ul>
<li><strong>性能与大规模场景优化</strong>：在保持 Redis 高性能特性的同时，面向大规模集群治理与工程化持续演进。</li>
<li><strong>安全迭代</strong>：社区模式下修复节奏与透明度更利于企业快速跟进。</li>
</ul>
<h3><a name="p-11583241-h-4-poc-11" class="anchor" href="https://linux.do#p-11583241-h-4-poc-11" aria-label="标题链接"></a>4. 企业实践背书（需要以实际案例与 PoC 为准）</h3>
<ul>
<li>Valkey 已被多家企业/云厂商纳入产品或托管服务（可参考 Valkey 官网 Participants 列表）。</li>
</ul>

<h1><a name="p-11583241-h-18" class="anchor" href="https://linux.do#p-11583241-h-18" aria-label="标题链接"></a>总结：</h1>
<ol>
<li>
<p><strong>Redis风险：</strong> <strong>许可证</strong> 和 <strong>协议</strong> 及 潜藏<strong>BUG</strong></p>
<ul>
<li><strong>协议 And 许可</strong>
<ul>
<li>Redis 新协议<strong>不符合 OSI 开源定义</strong>，对<strong>商业使用</strong>设置严苛限制</li>
<li><strong>SSPL 协议具有传染性</strong>，要求使用该协议代码的<strong>衍生作品也必须开源</strong></li>
<li>对<strong>法律合规</strong>要求极高，避免未来潜在诉讼</li>
</ul>
</li>
<li>潜藏<strong>BUG</strong> ：
<ul>
<li>潜伏 13 年的 Lua 脚本引擎 “释放后使用” 漏洞，CVSS 评分高达 10.0（最高危）</li>
<li>允许认证攻击者执行任意代码，控制服务器</li>
<li>全球约 6 万台 Redis 实例因未启用认证而完全暴露</li>
</ul>
</li>
</ul>
<p><strong>问题</strong> ：</p>
<ul>
<li>上述BUG已修复，但需要更新版本，但更新版本就需要开源代码</li>
<li>BUG众多，及已知BUG暂未修复，潜在BUG风险需谨慎考虑</li>
</ul>
</li>
<li>
<p>Redis平替： Valkey</p>
<p>​    目前国内众多公司也已经替换为Valkey ，包括但不限于<strong>字节，阿里云，腾讯云</strong>等大头</p>
<p>Valkey :</p>
<ul>
<li>免费且开源， 综合起来在各方面都比Redis强悍</li>
<li>社区资源丰富，生态好，且众多巨头公司参加Valkey 贡献（如上述<strong>字节、阿里云，还有谷歌，亚马逊</strong>等）</li>
</ul>
</li>
</ol>
