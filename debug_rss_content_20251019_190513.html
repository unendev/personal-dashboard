<h1><a name="p-9558428-api-model-settings-3" class="anchor" href="https://linux.do#p-9558428-api-model-settings-3"></a>二、API &amp; Model Settings</h1>
<h3><a name="p-9558428-v032sdk-api-model-settings-4" class="anchor" href="https://linux.do#p-9558428-v032sdk-api-model-settings-4"></a>在最新的<code>v0.3.2</code>版本中移除了所有官方SDK(依赖太重了),所以配置会有轻微区别，启动后进入 <code>API &amp; Model Settings</code> 可以看到如下内容</h3>
<ul>
<li><code>Profile</code> 用于切换或新建配置，Snow现在支持保存多套API和模型方案</li>
<li><code>Base URL</code> 请求地址，由于移除了官方SDK所以，OpenAI和Anthropic需要补<code>/v1</code>,Gemini需要补 <code>/v1beta</code></li>
<li><code>API Key</code> 没啥好说的，Key呗 <img src="https://linux.do/images/emoji/twemoji/distorted_face.png?v=14" title=":distorted_face:" class="emoji" alt=":distorted_face:" loading="lazy" width="20" height="20"></li>
<li><code>Request Method</code> 请求方案，按照你的需要选择：<code>Chat Completions</code>、<code>Responses</code>、<code>Gemini</code>、<code>Anthropic</code></li>
<li><code>Anthropic Beta</code> 这个复选框如果勾选，A÷的请求URL会自动补一个 <code>beta=true</code>的参数，方便大家兼容Claude Code API，当然在后续更新中，根据 A÷的调整，可能会取消</li>
<li><code>Advanced Model</code>、<code>Basic Model</code>、<code>Compact Model</code> 这三个分别设置：高级模型干活的、小模型总结的、压缩模型压缩上下文用的，这三个模型都来源于上面设置的 <code>BaseURL</code>和<code>API Key</code>,系统会自动请求对应的 <code>/models</code> 接口获取列表,支持键入过滤,当然遇到模型接口列表不完善的API,应用提供了 <code>Manual Input (Enter model name)</code>选项，可自行输入模型名称</li>
<li><code>Max Context Tokens</code> 模型的最大上下文窗口，用于计算上下文百分比用的，可以根据模型实际情况填写，例如 <code>gemini</code> 一般是 1M上下文，就填 1000000，这个参数不会影响实际的模型上下文，只会影响界面的计算结果</li>
<li><code>Max Tokens</code> 这个不能乱填，这个参数会直接加入API请求体中，作为 <code>max_tokens</code> 参数</li>
</ul>

<h1><a name="p-9558428-snow-14" class="anchor" href="https://linux.do#p-9558428-snow-14"></a>八、Snow的系统文件</h1>
<h3><a name="p-9558428-snow-snow-15" class="anchor" href="https://linux.do#p-9558428-snow-snow-15"></a>在用户目录的 <code>.snow</code> 文件夹中有Snow的所有文件，下面一个一个解释</h3>
<p></p><div class="lightbox-wrapper"><a class="lightbox" href="https://linux.do/uploads/default/original/4X/d/3/4/d34a89045d0490f8573b5cd6fe9f8f4b2a17cdbc.png" data-download-href="https://linux.do/uploads/default/d34a89045d0490f8573b5cd6fe9f8f4b2a17cdbc" title="image"><img src="https://linux.do/uploads/default/original/4X/d/3/4/d34a89045d0490f8573b5cd6fe9f8f4b2a17cdbc.png" alt="image" data-base62-sha1="u9ar6Li4wbYRPWUjJdbutsRaxek" width="690" height="378" data-dominant-color="FAFAF9"></a></div><p></p>
<ul>
<li><code>log</code> 该目录主要记录了运行日志，这些日志不会上传任何远端，仅留在本地用于出现BUG时方便定位，可任意删除</li>
<li><code>profiles</code> 前文中提到，Snow支持多个配置文件切换，用户自定义的多个配置文件都将保存在这个目录</li>
<li><code>sessions</code> 该目录保存了用户的所有对话记录，同样是用于 <code>/resume</code> 以及其他功能实现的必要目录，不会上传任何远端</li>
<li><code>snapshots</code> 该目录保存了AI自动编辑文件前的快照，用于文件回滚，一般情况无需查看该目录，一切都是自动的</li>
<li><code>todo</code> 该目录文件是每次对话的 <code>todo</code> 列表持久化，避免因为意外退出，而导致AI忘记任务的情况</li>
<li><code>active-profile.txt</code> 因为Snow的早期版本不支持多个配置文件，所以为了向下兼容，新建了一个文本文件，用于标识用户当前使用的是哪个配置文件</li>
<li><code>config.json</code> 无需多言，最重要的API配置信息</li>
<li><code>custom-headers.json</code> 自定义请求头的内容</li>
<li><code>mcp-config.json</code> 自定义MCP服务的内容</li>
<li><code>system-prompt.txt</code> 自定义系统提示词的内容</li>
</ul>

            <p>先叠个小甲，非代码专业人员，没啥专业基础，目前知识有限，有 理解偏差\错误 还望指出。真诚发问，各位程序员大佬，看到血压高请打它一拳出出气<img src="https://linux.do/images/emoji/twemoji/down_arrow.png?v=14" title=":down_arrow:" class="emoji" alt=":down_arrow:" loading="lazy" width="20" height="20"><br>

<p>佬友们刷信息流上瘾么？有没有只想搜什么东西，或者打开看看关注的up更新，就不知不觉刷起了推荐——刷推荐的时候还总希冀能刷到一些有趣有用的内容，在“浪费时间”和“害怕错过”之间反复受折磨。以上是我的情况 <img src="https://linux.do/images/emoji/twemoji/rofl.png?v=14" title=":rofl:" class="emoji" alt=":rofl:" loading="lazy" width="20" height="20"></p>

<ol>
<li>没有消息推送</li>
<li>订阅源不稳定：自己配置的源（比如b站知乎非热门博主的投稿）和官方实例（b站小约翰等热门up动态）均会出现不定时的失效。官方示例可能过几天会恢复，自建的不太会排查，经常前一天好好的，突然就不行了，手动更新cookie也只能挽救一部分，仍然有一些动不动就挂了。</li>
<li>RSSHub似乎不支持单个订阅源的精细管理，只能在 docker-compose.yml 文件中定义“缓存时间”这个全局参数，要多订阅不同缓存策略，只能部署多个实例。</li>
</ol>
<p>所以请教各位佬，有没有 消息推送+统一客户端+信息源稳定可自定义 的解决方案。（要求会不会太高了 <img src="https://linux.do/images/emoji/twemoji/melting_face.png?v=14" title=":melting_face:" class="emoji" alt=":melting_face:" loading="lazy" width="20" height="20"> 因为纯自用不需要啥规模 所以过反爬+稳定 应该是可以实现的吧？吧？ <img src="https://linux.do/images/emoji/twemoji/innocent.png?v=14" title=":innocent:" class="emoji" alt=":innocent:" loading="lazy" width="20" height="20">）</p>
<p>期望达到的效果：</p>
<ol>
<li>能够为不同的订阅源自定义方案（比如数据存储时间、刷新频率、防封反爬），比如“即时消息”能够主动推送 ；“例行浏览”能够定时更新。<br>
个人习惯是将信息源分两类：<br>
a. 和当下任务弱关联or无关联，但感兴趣。比如社交媒体频道<br>
希望能够定时推送，小时或者日级别，每天早上起来扫一眼<br>
b. 和当下任务强关联，必须“追新”的领域消息，比如官方消息通知<br>
希望能够即时推送，分秒级别</li>
<li>能够汇总到一个客户端，最好兼顾一下阅读时的体验。Reader wise配套的阅读器感觉就很好（但是感觉中文支持不太美丽，比如字符统计是摆设，中文源获取失败等等）。</li>
</ol>
<p>问问哈基米，得到了一些方案：<br>
1.Cron Job 使用 <code>cron</code>（Linux 的定时任务调度器）为不同的 RSSHub 链接设置不同的访问频率。<br>
——看起来这是解决“定时推送”最简单的方案，不知道有没有佬试过，和RSSHub适配如何。<br>
2.Webhook 通过自动化工具（如 n8n, Huginn, IFTTT, 或者自己写的脚本），监听上游服务的事件。一旦事件发生（例如，Telegram 频道收到新消息），自动化工具就立即调用一个接口或执行一个命令来访问对应的 RSSHub 链接，从而实现“零延迟”的缓存更新。<br>
——涉及到“push”和“轮询”等等专业领域，不是很懂。rss+n8n好像有不少用户，前者获取并处理信源数据，后者负责定时任务推送。还要再维护一套n8n容器系统么？<br>
3.通过其他的TTRSS、FreshRSS这样的自建客户端集中管理。<br>
4.找一些稳定专业的rss服务，哪怕付点钱呢——问题是，有覆盖国内外主流平台并对它们做相应优化的服务存在么？</p>
<p>除了第四条，以上方案均依托RSSHub这个“信息抓取和处理”的引擎，好像不太能解决订阅源时不时失效的问题，</p>

<h3><a name="p-9557940-rate-limits-1" class="anchor" href="https://linux.do#p-9557940-rate-limits-1"></a><strong>速率限制 (Rate limits)</strong></h3>
<p>速率限制规定了你在给定时间范围内可以向 Gemini API 发出的请求数量。这些限制有助于保持公平使用、防止滥用，并帮助维护所有用户的系统性能。</p>
<h3><a name="p-9557940-how-rate-limits-work-2" class="anchor" href="https://linux.do#p-9557940-how-rate-limits-work-2"></a><strong>速率限制如何运作 (How rate limits work)</strong></h3>
<p>速率限制通常从以下三个维度来衡量：</p>
<ul>
<li><strong>每分钟请求数 (RPM)</strong></li>
<li><strong>每分钟令牌数（输入）(TPM)</strong></li>
<li><strong>每日请求数 (RPD)</strong></li>
</ul>
<p>你的使用情况会根据每个限制进行评估，超过其中任何一个都会触发速率限制错误。例如，如果你的 RPM 限制是 20，那么在一分钟内发出 21 个请求就会导致错误，即使你没有超过 TPM 或其他限制。</p>
<p>速率限制是应用于 <strong>每个项目 (per project)</strong>，而不是每个 API 密钥。</p>
<p><strong>每日请求数 (RPD) 的配额会在太平洋时间午夜 (midnight Pacific time) 重置。</strong></p>
<p>限制会因所使用的具体模型而异，并且某些限制仅适用于特定模型。例如，“每分钟图像数 (Images per minute, IPM)”仅为能够生成图像的模型（如 Imagen 3）计算，但在概念上与 TPM 类似。其他模型可能有“每日令牌数限制 (token per day limit, TPD)”。</p>
<p>对于实验性和预览模型，速率限制会更加严格。</p>
